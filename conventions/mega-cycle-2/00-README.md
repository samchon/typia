# Mega-cycle 2 — 허점 공격

> 상태: 대기 중 (Mega-1 완료 후 진입)
> 입력: `../mega-cycle-1/01-boundary.md` ~ `06-release-dx.md` (Mega-1 의 Sub-5 최종본 6편)
> 산출: `mega-cycle-2/01-boundary.md` ~ `06-release-dx.md`

## Mega-2 의 고유 임무

Mega-1 의 확정본을 **모든 가능한 방식으로 공격**하여 허점·자기모순·약한 근거를 찾아 개정한다. Mega-1 은 "초안 체계"이고, Mega-2 는 그 체계가 실제로 견고한지 **파괴 시도**로 검증한다.

## Sub-1 (이번 mega 의 초안, 사실상 재작성)

**중요**: Mega-2 의 Sub-1 은 "새로 초안을 작성"하는 게 아니라, **Mega-1 최종본을 읽고 약점을 공격**하여 그 약점을 보완한 개정 초안을 낸다.

각 역할의 Sub-1 에이전트 프롬프트 요지:
- 입력: Mega-1 의 자기 역할 최종본 (`mega-cycle-1/0X-*.md`)
- 임무: **자기 초안을 자기가 공격** — "만약 내가 이 규약을 악용/회피하려면 어떤 식이 가능한가?" 3 가지 이상 찾아 반영.
- 타 역할의 Mega-1 최종본과 **경계 충돌** 재확인.
- 산출: `mega-cycle-2/cycles/cycle-01-X-redraft.md`

## Sub-2 (교차 리뷰) 특기 사항

Mega-1 과 달리 Mega-2 의 교차 리뷰는 다음을 추가 점검:
1. Mega-1 → Mega-2 변경 이유가 건전한가 (over-correction 여부)
2. Mega-1 에서 풀리지 않은 M1-Q* 질문이 여전히 미결이라면 왜
3. 규약 ID 충돌 (Mega-1 에서 쓰던 ID 를 Mega-2 가 바꿨다면 마이그레이션 가이드)

## Sub-3, 4, 5 — 표준 절차

Mega-1 과 동일한 흐름 (개정 → 감수 → 최종).

## 공격 체크리스트 (Sub-1 가이드)

에이전트에게 다음을 반드시 시도하도록 지시:
- 규약 위반이 CI 로 자동 감지되지 않는 경우 찾기
- 규약 인용 근거 (wiki/코드) 가 버전 bump 시 깨지는 시나리오
- 성능·재현 빌드 가정에 측정 근거 없이 기댄 부분
- 외부 레퍼런스 (tsgonest, tsgolint, effect-ts) 와 실제 상이한 부분
- samchon 개인 bandwidth 가정 (solo dev) 이 유지되는지
