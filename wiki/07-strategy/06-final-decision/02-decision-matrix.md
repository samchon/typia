## 그러나 수학만으로 결정하면 안 되는 이유

### 현실 변수 1 — 자원
C는 samchon 혼자면 22개월 풀타임. 그 동안 typia v12 유지보수 + nestia + AutoBE + Agentica 병행은 **번아웃 리스크 매우 큼** (R8).

### 현실 변수 2 — 조기 실패 비용
C는 M6~M7 (15개월+)에 v1.0. 그 전에 tsgonest가 선점하면 회복 어려움.

### 현실 변수 3 — 심리
TS 30K LOC 폐기는 심리적 비용. D는 둘 다 지원하므로 심리적 완충.

## 현실 반영 조정 (시나리오별)

### 시나리오 S1. samchon 혼자, 자원 제한
- C 단독: 리스크 너무 큼 → 3 → 2로 조정
- 결과: **A(LTS)+B(ttsc) 결합 = 옵션 B**가 최선

### 시나리오 S2. Go 조력자 1인 확보
- C와 D 모두 일정 단축
- C를 12~14개월로 달성 가능 → 가중치 수학 그대로 **C**

### 시나리오 S3. AutoBE 상업화 수익 확보
- D가 현실적 (두 트랙 동시 투자 가능)
- **D(Hybrid)**가 최선

## 시나리오별 권장

| 시나리오 | 최선 | 차선 |
|---|---|---|
| S1. 혼자, 자원 제한 | **B (ttsc)** | A 병행 |
| S2. Go 조력자 1인 | **C (typia-go)** | D |
| S3. 자원 충분 | **D (Hybrid)** | C |

## 의사결정 트리

```
Q1. Go 조력자 확보 가능? (Phase 1 최소 6개월)
├─ Yes
│  └─ Q2. 자원 여유 (AutoBE 수익 등)?
│      ├─ Yes → D (Hybrid)
│      └─ No → C (typia-go 집중)
└─ No
    └─ Q3. Go 학습 1~2개월 투자 의지?
        ├─ Yes → B (ttsc) 먼저, 추후 조력자 확보 시 C 전환
        └─ No → A (LTS 유지) + 장기 피벗 검토
```

## 각 옵션의 합격선 (pass criteria)

### A. ts-patch LTS
- TS 6.x 사용자 안정 유지
- 보안 패치만 지속
- 2028 말 graceful deprecation

### B. ttsc
- 12개월 내 v1.0
- tsc+ts-patch 대비 3× 이상 속도
- 매 tsgo release 1주 내 대응

### C. typia-go
- 18개월 내 v1.0 (22개월 허용)
- test-typia-automated 100%
- tsc+ts-patch 대비 8× 이상
- tsgonest 기능 parity + LLM/Protobuf 차별점 유지

### D. Hybrid
- Year 1: B v1.0
- Year 2: C alpha
- Year 3: C v1.0 + B deprecation

## 실패 시나리오

### 옵션별 최악 결과

- **A 단독 실패**: 2028년 사용자 -70%, typia deprecated
- **B 단독 실패**: IPC 오버헤드 통제 불가, 기대 성능 미달, tsgonest에 밀림
- **C 단독 실패**: 18개월 소비했는데 완성도 부족, 그 사이 tsgonest 장악
- **D 단독 실패**: 이중 투자로 번아웃, 둘 다 미완성

## 무엇을 포기해도 안 되는가

1. **사용자 API** — 어떤 옵션이든 `typia.is<T>(input)` 불변
2. **LLM function calling** — typia 해자, C에서 우선 이식
3. **13 namespace 범위** — 일부 v1.1로 미루되 포기는 X
4. **nestia 호환** — 나쁘게든 좋게든 호환 유지
